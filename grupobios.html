Reporte de Arquitectura Técnica

Análisis, Evaluación y Propuestas de Mejora
Cliente: Grupo Bios - Sistema de Laboratorios Clínicos
Elaborado por: Lenosoft
Fecha: Noviembre 2025

1. Entendimiento de la Arquitectura Actual

1.1. Contexto General

El sistema corresponde a una plataforma para laboratorios clínicos, que integra información proveniente de equipamiento médico y de laboratorios (resultados de exámenes, datos de pacientes, etc.). Se trata de un sistema core clínico, donde seguridad, confidencialidad, trazabilidad y disponibilidad son críticas.

1.2. Stack Tecnológico Declarado

Front-end

Framework: Next.js (sobre React).

Rol: Interfaz de usuario para profesionales de la salud y/o personal administrativo, consumo de API REST del backend para:

  • Visualización de resultados de laboratorio
  • Interacción con fichas de pacientes
  • Gestión operativa del laboratorio
Back-end / API

Tecnología: Java + Spring (versión reportada como obsoleta por el equipo).

Empaquetado: Stack clásico, servicios empaquetados en un WAR.

Servidor de aplicaciones: Tomcat.

Arquitectura lógica: Monolito con módulos internos, organizados dentro de una misma aplicación desplegada como un único WAR.

Exposición de servicios: API REST usada por el front-end y posiblemente por integraciones externas.

Documentación de API: Actualmente la API no cuenta con documentación formal (no se utiliza Swagger / OpenAPI).

Base de Datos

Motor: Oracle.

Capa de persistencia: Hibernate como ORM.

El equipo declara problemas con Hibernate, probablemente relacionados con rendimiento, mapeos complejos, gestión de transacciones o compatibilidad de versiones (requiere análisis más fino).

DevOps / CI/CD

Repositorio de código: Bitbucket.

Pipelines: Bitbucket Pipelines para:

  • Compilación y empaquetado del WAR
  • Despliegue al servidor de aplicaciones correspondiente

Testing automatizado: De acuerdo a lo levantado, no cuentan con tests automatizados (unitarios, de integración ni end-to-end).

Seguridad

Mecanismo de autenticación: Esquema propio basado en JSON Web Token (JWT).

Autorización y perfilamiento:

  • Se mencionan temas de perfilamiento aún en desarrollo o incompletos (roles y permisos no totalmente modelados o aplicados).
  • Existe preocupación explícita por una posible falta de control en Broken Object Level Authorization (BOLA) según OWASP, es decir, riesgo de que un usuario pueda acceder a recursos (exámenes, fichas, etc.) que no le corresponden, manipulando identificadores o endpoints.
Observabilidad

El cliente declara que no tienen herramientas de observabilidad implementadas:

  • Logs dispersos y no centralizados
  • Sin métricas de aplicación ni trazas distribuidas
  • Sin paneles de monitoreo ni alertas formales

Situación de versiones (Spring / Java / Hibernate)

El equipo técnico declara que la versión de Spring está obsoleta y que uno de sus objetivos es actualizar la versión de Java.

Actualmente el ecosistema se mueve hacia:

  • Spring Framework en ramas 6.x/7.x
  • Java LTS en 17/21 en adelante

Estar en versiones antiguas de Spring/Hibernate implica:

  • Riesgo de seguridad (vulnerabilidades no parchadas)
  • Compatibilidad limitada con librerías modernas
  • Dificultad para introducir mejores prácticas (observabilidad, resiliencia, etc.)

2. Riesgos y Puntos de Atención Identificados

API REST sin documentación formal

Dificulta la incorporación de nuevos desarrolladores, las integraciones, la automatización de pruebas y la gobernanza de servicios.

Ausencia de testing automatizado

Cada cambio en el código incrementa la probabilidad de regresiones. No hay una red de seguridad que valide flujos críticos (ingreso de resultados, cálculo de informes, etc.).

Autenticación propia y autorización potencialmente débil (BOLA)

El uso de JWT propio, sin un estándar claramente definido (OAuth2/OpenID Connect), sumado a problemas de perfilamiento, aumenta el riesgo de:

  • Acceso indebido a datos de pacientes
  • Incumplimiento de OWASP Top 10 (especialmente Broken Object Level Authorization)
  • Problemas regulatorios y reputacionales si se produce una brecha
Monolito en WAR sobre Tomcat

No es necesariamente "malo", pero:

  • Limita la flexibilidad para escalar componentes específicos
  • Aumenta el impacto de cambios y despliegues (riesgo de "todo o nada")
  • Dificulta la adopción de patrones modernos (deployment independiente por contexto de negocio, contenedores, etc.)
Problemas declarados con Hibernate

Pueden traducirse en:

  • Rendimiento deficiente en consultas complejas
  • Bloqueos o problemas de concurrencia
  • Complejidad para evolucionar el modelo de datos
Falta de observabilidad
  • Difícil investigar incidentes
  • Limitada capacidad para medir SLA (tiempo de respuesta, disponibilidad)
  • Poca visibilidad para mejorar proactivamente la plataforma
Perfilamiento incompleto

En un contexto clínico, se requieren perfiles y permisos muy específicos (médico, tecnólogo médico, administrativo, auditor, etc.), con reglas claras de acceso a datos sensibles.

3. Auto-Assessment de la Arquitectura

Esta sección puede presentarse como checklist o formulario de autoevaluación, con escala 1–5 (1 = muy débil, 5 = muy sólido).

3.1. Front-end (Next.js)

Modelado de UI

En una escala de 1 a 5, ¿cómo evalúas la consistencia del diseño de la interfaz (componentes reutilizables, design system, coherencia visual entre pantallas)?

Estado y lógica en el front

¿Qué tan claro y ordenado está el manejo del estado (formularios, errores, datos de API) en el front-end?

Seguridad en el front

¿En qué medida el front implementa prácticas de seguridad básicas (no exponer tokens en storage inseguro, protección frente a XSS, CSRF, etc.)?

3.2. Backend / API (Java + Spring + Tomcat)

Organización del código

¿Cómo evaluarías la claridad de la separación por capas (controller, service, repository) dentro del monolito?

Consistencia de la API

¿Qué tan homogéneo es el diseño de los endpoints (nomenclatura, uso de verbos HTTP, códigos de respuesta)?

Evolutividad del monolito

¿En qué medida el monolito actual permite incorporar nuevas funcionalidades sin generar efectos colaterales difíciles de controlar?

3.3. Seguridad y cumplimiento (JWT, BOLA, perfilamiento)

Gestión de JWT

¿Qué tan robusta es la gestión de tokens (generación, expiración, revocación, almacenamiento seguro de claves)?

Autorización por objeto (BOLA)

¿En qué nivel sienten que hoy existe un control sistemático para asegurar que cada usuario solo accede a exámenes y fichas para las cuales tiene permiso?

Perfilamiento de usuarios

¿Qué tan bien definidos y aplicados están los roles y permisos según el tipo de usuario (médico, tecnólogo, administrativo, etc.)?

Cumplimiento regulatorio

¿En qué medida el sistema actual satisface requisitos de privacidad, trazabilidad y acceso a datos clínicos exigidos por la normativa aplicable?

3.4. Datos y persistencia (Oracle + Hibernate)

Modelo de datos

¿Hasta qué punto el modelo de datos está documentado y alineado con el modelo de negocio actual?

Rendimiento de consultas

¿Cómo evalúan el rendimiento de las consultas más utilizadas (tiempos de respuesta, bloqueos, escalabilidad)?

Gestión de cambios de esquema

¿Qué tan controlado y trazable es el proceso de cambiar el esquema de la base de datos (scripts manuales vs herramientas como Liquibase/Flyway)?

3.5. CI/CD y calidad

Pipeline de integración continua

¿En qué medida el pipeline de Bitbucket automatiza las tareas clave (build, análisis de código, despliegue en ambientes)?

Pruebas automatizadas

Hoy no hay tests automatizados: ¿qué tan urgente consideran incorporar al menos un set mínimo de pruebas unitarias y de integración?

Gestión de despliegues y rollback

¿Qué tan definido está el proceso para desplegar en producción y, si es necesario, volver rápidamente a una versión anterior?

3.6. Observabilidad y operación

Centralización de logs

¿Qué tan fácil es hoy rastrear un incidente extremo a extremo (desde el request del front, pasando por el backend, hasta la base de datos)?

Métricas y alertas

¿En qué medida cuentan con métricas y alertas que les permitan anticipar problemas antes de que afecten a los usuarios?

Monitoreo de servicios críticos

¿Qué tan visible es el estado de los procesos clínicamente críticos (por ejemplo, recepción de resultados desde equipos, publicación de resultados a médicos/pacientes)?

4. Propuestas de Mejora sobre la Arquitectura Actual

4.1. Documentación y gobernanza de API

Propuesta
  • Introducir OpenAPI/Swagger para documentar la API REST
  • Establecer convenciones de diseño de API (versionado, códigos HTTP, manejo estándar de errores)
  • Facilitar la creación de clientes y automatización de pruebas de contrato

4.2. Seguridad: autenticación, autorización y BOLA

Fortalecer el modelo de autorización por objeto
  • Implementar una capa centralizada para verificar propiedad/autorización antes de acceder a recursos (exámenes, fichas, resultados)
  • Definir patrones claros (servicios de autorización, anotaciones, interceptores)
Revisar el esquema de JWT propio
  • Asegurar almacenamiento y rotación segura de claves
  • Implementar revocación, expiración y scopes/claims claros
Evaluar transición a un IdP estándar
  • Considerar OAuth2/OpenID Connect con proveedores como Keycloak, Auth0, Azure AD u otros
  • Dejar el back-end responsable solo de validar tokens y aplicar reglas de autorización de negocio
Revisiones de seguridad periódicas
  • Integrar análisis estático y dinámico en el pipeline (SAST/DAST)
  • Revisar específicamente OWASP Top 10 (con foco en BOLA, gestión de sesiones, exposición de datos sensibles)

4.3. Modernización del stack (Spring / Hibernate / empaquetado)

Plan de migración
  • Migración de Spring a una versión soportada y moderna
  • Mantener Java 17 (LTS) o preparar el terreno para una futura migración a versiones LTS más nuevas
  • Revisar la versión de Hibernate: ajustar configuración de fetch, caché e índices de Oracle
  • Incorporar herramientas de versionado de esquema (Liquibase/Flyway)
Evaluar a mediano plazo
  • Migrar desde WAR en Tomcat hacia Spring Boot jars (con o sin contenedores Docker)
  • Mantener el monolito pero modularizarlo mejor internamente, o ir abriendo ciertos módulos en servicios separados cuando tenga sentido de negocio

4.4. Testing y calidad

Estrategia incremental de tests
  • Tests unitarios sobre lógica de negocio crítica
  • Tests de integración (Spring + Oracle en ambiente controlado)
  • Tests end-to-end sobre los flujos clínicos más relevantes
  • Incorporar los tests al pipeline de Bitbucket: que el build falle si se rompen tests clave
  • Reportes de cobertura

4.5. Observabilidad

Implementar un stack mínimo de observabilidad
  • Logs estructurados y centralizados (por ejemplo ELK, Loki+Promtail, o la solución que encaje con su infraestructura)
  • Correlación de requests (correlation-id)
  • Métricas de aplicación (latencia, error rate, throughput)
  • Alertas sobre: caídas de servicio, aumento de errores 5xx, lags en la recepción/envío de resultados de laboratorio

5. Ideas de Nuevas Funcionalidades y Productos con IA

5.1. "Copiloto clínico" sobre la ficha del paciente

Un asistente conversacional para el médico que permita hacer preguntas del tipo:

  • "Resúmeme la evolución de la creatinina de este paciente en los últimos 5 años"
  • "¿Qué exámenes relevantes tiene relacionados con función hepática?"
  • "¿Este paciente ha tenido resultados anómalos repetidos en los últimos 12 meses?"
Técnica:
  • Ingesta de resultados históricos, diagnósticos y notas (cuando existan)
  • Vector store + RAG para consultas semánticas
  • Resúmenes temporales (timeline clínico)

5.2. Priorización inteligente de casos y resultados críticos

Modelo que identifique resultados críticos (valores que implican riesgo vital o requieren acción urgente) y:

  • Priorice la visualización al médico
  • Genere alertas al equipo clínico o call center

Puede combinar reglas clínicas explícitas (rangos críticos) con modelos que detecten combinaciones de exámenes asociadas a mayor riesgo (ej. sepsis, falla renal aguda, etc.).

5.3. Resumen automático de la historia de laboratorio

Generar automáticamente un "Resumen de laboratorio" por paciente:

  • Principales diagnósticos relacionados a exámenes
  • Tendencias (mejora/empeoramiento) de marcadores relevantes (HbA1c, creatinina, perfil lipídico, etc.)
  • Alertas de adherencia (ej. "el paciente no ha repetido este examen clave en 12 meses")

Esto es especialmente útil para pacientes con mucha data, donde el médico hoy pierde tiempo navegando múltiples pantallas.

5.4. Explicaciones amigables de resultados para pacientes

Generar informes en lenguaje natural, comprensible para el paciente:

  • "Tu colesterol LDL está alto. Esto significa que…"
  • "Se recomienda conversar con tu médico sobre…"

Siempre dejando claro que es información educativa, no diagnóstico. Puede conectarse al portal del paciente o a reportes PDF.

5.5. Análisis de eficiencia y calidad del laboratorio

Usar IA para analizar:

  • Tiempos de procesamiento de muestras
  • Retrabajos, exámenes repetidos
  • Desviaciones en series temporales de resultados (control de calidad interno)

Se pueden construir dashboards de productividad y calidad para la gestión del laboratorio.

5.6. Asistente para el tecnólogo médico / personal del lab

Un agente interno que responda sobre:

  • Protocolos del laboratorio (ej. preparación de paciente, manejo de muestras)
  • SOPs (procedimientos operativos estándar)
  • Manuales de equipos, ranges de control, etc.

Técnicamente, un RAG sobre documentos internos (PDF, manuales, protocolos, hojas de cálculo de control de calidad).

5.7. Detección de patrones en resultados longitudinales

Modelos que identifiquen patrones en tiempo:

  • Pacientes cuya curva de ciertos marcadores va en deterioro, aunque aún dentro del rango "normal"
  • Riesgo de progresión de enfermedades crónicas en base a historial de laboratorio + demografía

Esto se puede plantear al inicio más como analítica avanzada (no como diagnóstico), apoyando programas preventivos.

5.8. Integración con imágenes (si el laboratorio procesa imágenes)

Si en el futuro integran imágenes (p.ej., hematología, anatomía patológica, slides, etc.), se puede:

  • Usar modelos de visión por computador para ayudar en el pre-screening (ej. priorizar láminas con mayor probabilidad de hallazgos)
  • Siempre con fuerte supervisión humana y sin reemplazar el dictamen del especialista
